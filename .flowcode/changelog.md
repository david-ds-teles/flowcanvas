# Changelog

- The single record of what changed in each flowcode version — readable history for the team **and** the migrator's transform carrier.
- One entry per framework `version` (the `version` in `flowcode.yml`), newest on top under `## Log`.
- **Every** version entry carries (1) a human summary, and (2) a `<!-- files:auto -->` block listing every framework file Added / Changed / Removed in that version. **The file list is auto-generated by `bundle.js`** from the diff between this version's `framework-manifest.json` and the previous version's — never hand-edited. `bundle.js` **fails closed** if the list and the manifest disagree, so an updated file can never be missing from the changelog.
- An entry **also** carries a `**Migration**` block **only** when its change is *non-inferable* — a rename, move, removal, or restructure of **host-owned** content the migrator cannot derive from the file delta alone.
- **How `/flowcode:migrate` uses this file:** the file-level delta (which framework files to copy/remove) comes from `framework-manifest.json`, computed by `migrate-plan.js` — *not* parsed from this prose. The migrator reads only the `**Migration**` blocks of versions newer than the installed `version`, and applies each block's `Action`. Summary-only / file-list-only entries are inferable — the delta engine handles them; they are not actionable by the agent.
- Append-only for shipped entries — never edit or delete history (except the auto `files:auto` block, which `bundle.js` regenerates). Correct a mistake with a new entry under a new version.
- Not the per-plan `{PREFIX}-changelog` (a plan deliverable under `plans/{PREFIX}/`, generated from `templates/changelog-template.md`). This file is the framework's own changelog at the `.flowcode/` root.

## When to Add an Entry

- **Every version bump** gets an entry with a human summary describing what changed and why. The `files:auto` Added/Changed/Removed block is filled in automatically by `bundle.js` — do not write it by hand.
- Add a `**Migration**` block to that entry **only** when a change would silently strand or corrupt host-owned content unless the migrator is told how to transform it:
  - **rename** — a file, directory, section heading, or frontmatter key changed name, and host files reference the old name.
  - **move** — a file or its content moved to a new path/parent, and host installs hold it at the old location.
  - **remove** — a framework artifact (template, hook, command, rule) was deleted, and stale copies must be pruned from installs.
  - **restructure** — a cross-cutting format change (e.g. a new mandatory frontmatter block, an index-format switch) every affected host file must be reshaped to match.
- Do **not** add a `**Migration**` block for additive-only changes (new file, new optional field) or anything the migrator already does by default (overwrite framework files byte-for-byte, backfill frontmatter/summary). Those are inferable — a summary-only entry is correct, and a noise block makes the migrator do redundant work.

## Entry Format

Each entry is a level-2 section headed by the introducing framework version and date:

```markdown
## <version> — <short title>   ·   <YYYY-MM-DD>

<Human summary: what changed and why. Group as Added / Changed / Fixed / Removed /
Breaking when useful. Present on every entry.>

**Migration** · Type: rename | move | remove | restructure
From: <observable old state host installs currently hold>
To:   <observable new state — the current spec>
Action: <exactly what the migrator must do to a host-owned file to converge it; cite
concrete paths and headings; preserves host body content>
```

- The `files:auto` block is written by `bundle.js` from the manifest diff — author the human summary and any `**Migration**` block; leave the file list to the build.
- `<version>` is the `version` in `flowcode/flowcode.yml` (semver) the change shipped in. The migrator applies a `**Migration**` block only to installs older than this version.
- `Type` may list more than one keyword when a single shipped change is genuinely multi-faceted (e.g. `restructure` that also `rename`s a heading).
- A version with several non-inferable changes carries several `**Migration**` blocks, one per change.
- `From` / `To` describe the *observable* old and new state, not the rationale.
- `Action` is migrator-executable: name the files/headings/keys to touch and the transform to apply. It must preserve host body content.

## 0.7.0 — Changelog/manifest-driven delta migration   ·   2026-06-25

Changed:
- **`/flowcode:migrate` is now a delta engine, not a per-file sweep.** A new dependency-free `migrate-plan.js` diffs the installed `install-manifest.json` against the framework's shipped `framework-manifest.json` (per-file sha256) and touches **only** the framework files that actually changed/were added/removed — copying them with a script. The old behavior (an LLM sub-agent that hashed and reasoned over *every* shipped file, then swept every host-owned file) was slow and token-heavy; it survives **only** as a `full-convergence` fallback for legacy/pre-baseline installs (no `version`/`sha256` in the manifest, or a source shipping no `framework-manifest.json`).
- **The common upgrade runs with no LLM at all.** The command runs `migrate-plan.js`, and dispatches the `flowcode:migrator-agent` sub-agent **only** when there is judgment work: a host-edited framework file to harvest, or an applied version carrying a `**Migration**` block. Otherwise it copies the changed files, merges hooks, restamps the manifest, and reports — directly.
- **Delta is computed over framework-owned files only** (filtered via `isHostOwned`); the script can never remove or overwrite host work. Host-edited framework files in the delta are harvested as `UC-NNN` before overwrite/removal, as before.

Added:
- **`framework-manifest.json`** — shipped per-version checksum manifest of framework-owned files (installed-path form); the machine source of truth for the delta. Generated + self-checked by `bundle.js`.
- **Auto file-list in every changelog entry** — `bundle.js` writes the `Added/Changed/Removed` block (the `files:auto` markers) from the same manifest diff and fails the build if it and the manifest disagree, so every updated file is recorded here.
- Shared `installedPathFor` / `mergeHookRegistrations` / dev-only set in `install-lib.js`, used by the installer, bundler, and migrator so all three agree on path mapping and hook merging.

All changes are framework-owned (the migrate command/agent, the new manifest, install-engine internals). No host-owned content needs transforming, so there is **no `**Migration**` block** — the delta engine installs the new files and the next migrate restamps the manifest with the new per-file checksums automatically.
