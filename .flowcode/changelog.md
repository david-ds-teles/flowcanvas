# Changelog

- The single record of what changed in each flowcode version — readable history for the team **and** the migrator's transform carrier.
- One entry per framework `version` (the `version` in `flowcode.yml`), newest on top under `## Log`.
- **Every** version entry carries (1) a human summary, and (2) a `<!-- files:auto -->` block listing every framework file Added / Changed / Removed in that version. **The file list is auto-generated by `bundle.js`** from the diff between this version's `framework-manifest.json` and the previous version's — never hand-edited. `bundle.js` **fails closed** if the list and the manifest disagree, so an updated file can never be missing from the changelog.
- An entry **also** carries a `**Migration**` block **only** when its change is *non-inferable* — a rename, move, removal, or restructure of **host-owned** content the migrator cannot derive from the file delta alone.
- **How `/flowcode:migrate` uses this file:** the file-level delta (which framework files to copy/remove) comes from `framework-manifest.json`, computed by `migrate-plan.js` — *not* parsed from this prose. The migrator reads only the `**Migration**` blocks of versions newer than the installed `version`, and applies each block's `Action`. Summary-only / file-list-only entries are inferable — the delta engine handles them; they are not actionable by the agent.
- Append-only for shipped entries — never edit or delete history (except the auto `files:auto` block, which `bundle.js` regenerates). Correct a mistake with a new entry under a new version.
- Not the per-plan `{PREFIX}-changelog` (a plan deliverable under `plans/{PREFIX}/`, generated from `templates/changelog-template.md`). This file is the framework's own changelog at the `.flowcode/` root.

## When to Add an Entry

- **Every version bump** gets an entry with a human summary describing what changed and why. The `files:auto` Added/Changed/Removed block is filled in automatically by `bundle.js` — do not write it by hand.
- Add a `**Migration**` block to that entry **only** when a change would silently strand or corrupt host-owned content unless the migrator is told how to transform it:
  - **rename** — a file, directory, section heading, or frontmatter key changed name, and host files reference the old name.
  - **move** — a file or its content moved to a new path/parent, and host installs hold it at the old location.
  - **remove** — a framework artifact (template, hook, command, rule) was deleted, and stale copies must be pruned from installs.
  - **restructure** — a cross-cutting format change (e.g. a new mandatory frontmatter block, an index-format switch) every affected host file must be reshaped to match.
- Do **not** add a `**Migration**` block for additive-only changes (new file, new optional field) or anything the migrator already does by default (overwrite framework files byte-for-byte, backfill frontmatter/summary). Those are inferable — a summary-only entry is correct, and a noise block makes the migrator do redundant work.

## Entry Format

Each entry is a level-2 section headed by the introducing framework version and date:

```markdown
## <version> — <short title>   ·   <YYYY-MM-DD>

<Human summary: what changed and why. Group as Added / Changed / Fixed / Removed /
Breaking when useful. Present on every entry.>

**Migration** · Type: rename | move | remove | restructure
From: <observable old state host installs currently hold>
To:   <observable new state — the current spec>
Action: <exactly what the migrator must do to a host-owned file to converge it; cite
concrete paths and headings; preserves host body content>
```

- The `files:auto` block is written by `bundle.js` from the manifest diff — author the human summary and any `**Migration**` block; leave the file list to the build.
- `<version>` is the `version` in `flowcode/flowcode.yml` (semver) the change shipped in. The migrator applies a `**Migration**` block only to installs older than this version.
- `Type` may list more than one keyword when a single shipped change is genuinely multi-faceted (e.g. `restructure` that also `rename`s a heading).
- A version with several non-inferable changes carries several `**Migration**` blocks, one per change.
- `From` / `To` describe the *observable* old and new state, not the rationale.
- `Action` is migrator-executable: name the files/headings/keys to touch and the transform to apply. It must preserve host body content.

Changed: each lifecycle phase (research · design · plan · execute · bootstrap) is now an independently invocable skill + command, and every agent's frontmatter `name:` carries an `-agent` suffix (matching its filename) so the bare verb is freed for the skill layer. `/flowcode:brainstorm` becomes a thin alias into the `design` skill.

Added: five `skills/flowcode/{research,design,plan,execute,bootstrap}/SKILL.md` and commands `research.md` · `design.md` · `plan.md` · `execute.md`.

**Migration** · Type: rename
From: Pre-0.2.0 — sub-agent dispatch handles are the bare role (`flowcode:researcher`, `flowcode:designer`, `flowcode:planner`, `flowcode:code-explorer`, `flowcode:code-reviewer`, `flowcode:qa-runner`, `flowcode:artifact-updater`, `flowcode:bootstrap`, `flowcode:migrator`). The lifecycle phases have no standalone skill+command surface — design lives only inside `/flowcode:brainstorm`; research, plan, and execute live only inside agents or the orchestrator.
To: Current — every agent's frontmatter `name:` carries an `-agent` suffix (`flowcode:designer-agent`, …), matching its filename and freeing the bare verb for the skill layer. Each phase is now an independently invocable skill+command: `flowcode:research` · `design` · `plan` · `execute` · `bootstrap` skills at `.claude/skills/flowcode/{phase}/SKILL.md`, with commands `/flowcode:research` · `/flowcode:design` · `/flowcode:plan` · `/flowcode:execute` (`/flowcode:brainstorm` is now a thin alias into the `design` skill; `/flowcode:bootstrap` runs the `bootstrap` skill).
Action: (1) **Overwrite** all framework-owned files (agents, commands, workflow, plan-instructions, existing skills) — they already carry the new `-agent` handles. (2) **Add** the additive framework files — the five `skills/flowcode/{research,design,plan,execute,bootstrap}/SKILL.md` and commands `research.md` · `design.md` · `plan.md` · `execute.md`. (3) **Rename in host-owned/custom files only:** in any host-authored `.md` that dispatches an agent by its bare handle, rewrite `flowcode:<role>` → `flowcode:<role>-agent` for role ∈ {researcher, designer, planner, code-explorer, code-reviewer, qa-runner, artifact-updater, bootstrap, migrator}. Leave slash commands (`/flowcode:bootstrap`) and the new skill names (`flowcode:research`/`design`/`plan`/`execute`/`bootstrap`) untouched. Standard host installs reference agents only through framework files, so this step is usually a no-op.

Changed: `ui/` splits into framework-owned guidance (`ui-index.md` · `ui-workflow.md` · `ui-mockup-discipline.md`) and host-owned design ground truth (`ui/ui-design-system.md`, `ui/references/`, `ui/mockups/`).

**Migration** · Type: restructure | move

There is no `ui-design-system.md`, no `ui/references/`; the UI gate dispatches a `frontend-design` skill that was never defined.
To: Current — `ui/` is split: `ui-index.md` · `ui-workflow.md` · `ui-mockup-discipline.md` are framework-owned; `ui/ui-design-system.md`, `ui/references/`, and `ui/mockups/` are HOST-OWNED. Mockups are produced by the `flowcode:ui-mockups` composer skill (`.claude/skills/flowcode/ui-mockups/`, with vendored taste lenses under `references/taste/`), exposed standalone as `/flowcode:mockup`. New: `/flowcode:render-html`, templates `ui-design-system-template.md` + `html-deliverable-template.html`, and a bootstrap UI design-system harvest step.
Action: (1) **Re-classify `ui/`** — where a host install links/holds `ui/` as a single framework unit, convert to the three individual framework files (`ui-index.md`, `ui-workflow.md`, `ui-mockup-discipline.md`) and **seed** `ui/ui-design-system.md` + `ui/references/` from the framework source if absent, then treat them as host-owned (backfill frontmatter/summary only, never overwrite a harvested system). (2) **Overwrite** the three framework `ui/*` files and **write the new framework files**: the `ui-mockups` skill directory (incl. `references/taste/` + `house-style.css` + `quality-checklist.md`), commands `mockup.md` + `render-html.md`, templates `ui-design-system-template.md` + `html-deliverable-template.html`. (3) For frontend projects whose `ui-design-system.md` is still the verbatim starter, flag in the report that it should be harvested via `flowcode:bootstrap-agent § Step 6.5`. (4) No new hook registrations — the existing `frontmatter-summary-check.js` is overwritten (it now exempts `skills/flowcode/**/references/` so vendored taste docs pass).

## 0.0.0 — Frontmatter+summary standard, file-conventions, migrate & feedback subsystems (baseline)   ·   2026-06-23

Changed (baseline): every managed `.md` carries the 5-key frontmatter (`name, description, status, tags, links`) + a ≤10-bullet summary (`workflow/file-conventions.md`); `*-index.md` files become file-listing-only; the install manifest gains `version` + per-file `sha256`.

Added: `/flowcode:migrate` as the supported upgrade path; the frontmatter and feedback-capture hooks.

**Migration** · Type: restructure
From: Pre-1.0 installs — managed `.md` files carry no YAML frontmatter and no bullet summary; `*-index.md` files mix lifecycle rules/procedures with file rows; the only upgrade path is re-running `flowcode.sh`; the install manifest has no `version` or per-file `sha256`; no feedback-capture hook is registered.
To: Current spec — every managed `.md` under `.flowcode/` and `.claude/{agents,commands,skills,rules}/flowcode/` carries the 5-key frontmatter (`name, description, status, tags, links`) + a ≤10-bullet summary (see `workflow/file-conventions.md`); `*-index.md` files are file-listing-only (brief opener + file-row table); `/flowcode:migrate` (this agent) is the supported upgrade path; the manifest is stamped with `version` + per-file `sha256`; the frontmatter and feedback hooks are registered.
Action: This is the **baseline** entry — an install whose manifest has no `version`/`sha256` field is *pre-baseline* and must be fully converged: (1) backfill the 5-key frontmatter + ≤10-bullet summary into every host-owned `.md` (`plans/` artifacts + `plan-index.md`, `project/*`, `quality-checks/*`, `researches/*`, `upstream-contributions.md`), preserving all body content; (2) reshape every host-owned `*-index.md` to the file-listing-only format, lifting any lifecycle rules out of the index (they live in the file the agent loads, not the index); (3) overwrite all framework-owned files; (4) harvest any host edits to framework files as `UC-NNN` rows in `upstream-contributions.md` before overwriting; (5) merge the new hook registrations (`frontmatter-summary-check.js`, `feedback-nudge.js`) into `.claude/settings.json`; (6) restamp the manifest with `version` `0.0.0` and per-file `sha256`. Subsequent upgrades compare against this baseline via the recorded checksums.
