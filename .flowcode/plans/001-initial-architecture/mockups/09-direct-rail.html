Architecture Decision

Flowcanvas treats the board itself as the medium. A markdown file is no longer a flat document with paths buried in its frontmatter — it is a node whose relationships are drawn, labeled, and navigable.

This decision pins the boundary between persisted JSONCanvas and the rendered graph to exactly one place, so every other module can stay pure.

Adapter boundary

The adapter maps a FlowcanvasDoc onto React Flow nodes and edges. It is the only code allowed to translate coordinates or resolve color presets.

  • A pure, side-effect-free transform — trivially unit-testable.
  • Edges are derived from each file's links: frontmatter, never stored twice.
  • JSONCanvas presets "1".."6" resolve through colorVar() and nowhere else.
Keep the seam thin: if a module needs to know a node's pixel position, it asks the adapter — it does not reach into the document.

toReactFlow()

The reference implementation stays small enough to read in one screen:

// lib/canvas/adapter.ts
export function toReactFlow(doc: FlowcanvasDoc) {
  const nodes = doc.nodes.map((n) => ({
    id: n.id,
    type: nodeKind(n),
    position: { x: n.x, y: n.y },
    width: n.width ?? 320,
    height: n.height ?? 180,
    data: { color: colorVar(n.color) },
  }))
  return { nodes, edges: deriveEdges(doc) }
}

Syntax highlighting (shiki) runs only here in the reader — node bodies use a lighter client renderer.
