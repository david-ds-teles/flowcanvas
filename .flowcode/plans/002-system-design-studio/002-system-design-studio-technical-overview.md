# Technical Overview — 002-system-design-studio Flowcanvas System Design Studio (v2)

- Promotes `.canvas` from a layout/edge store into the single system-of-record for the design graph — typed/labeled relationships, agent-creatable groups, MCP-native round-trip, snapshot-diff change-review, template library, bundle export, and the tri-pane studio shell.
- Status complete; dated 2026-06-28; generated by the post-execution pipeline (sonnet, code-verified). Full spec conformance confirmed by the Code Explorer audit with 8 intended divergences documented below.
- Role: authoritative source for `002-system-design-studio-changelog.md` and `002-system-design-studio-test-notes.md`; reconciled against code before generation.
- Source plan: `002-system-design-studio-plan.md`.

---

## Summary

**What:** A full v2 System Design Studio built on top of Flowcanvas v0.2 — the `.canvas` file is now the authoritative typed-relation graph (content stays in live `.md` files), an MCP sidecar exposes the board to any MCP-capable agent harness over 7 typed tools, a snapshot-diff change-review surface gates every agent round (accept/discard), a template library scaffolds reusable node/diagram/document fragments, and a tri-pane workbench shell (structure rail · canvas · inspector) wraps the existing diagram-first canvas with collapsible, context-aware side panels.

**Why:** The v0.1 canvas was a markdown viewer board: edges were file-only, derived from `links:` frontmatter, with no typed semantics and no agent wire beyond clipboard copy/paste. The v2 design removes the multi-store merge tax — one authoritative canvas graph enables typed cross-endpoint linking, agent-driven import via the extraction spec, native MCP round-trip, navigable change-review, and portable bundle export without a second source to reconcile.

**Risks:** MCP handshake is integration-level (needs a running app + an MCP client); not covered by the automated test suite. The disk-divergence reconcile banner is unbuilt (the `resyncFile` action exists at `lib/canvas/store.ts:265`, but the banner/trigger detection are deferred). `get_board` does not stamp `session.lastBriefId`, so `apply_response` reports `stale:true` (advisory; merge is last-writer-wins and correct). `frontmatter links:` fields are treated as root-relative (not resolved against `basePath`), matching the plan's own worked example but diverging from the spec text.

**Tech-Debt:** Visual-parity pixel capture at 1280/1440 is deferred (no unattended browser-automation harness). Live MCP-status probe in the inspector Submit mode is still an info-level follow-up (the pill is currently static "MCP ready"). `reviewDiff().files` is derived from added file-node paths (MCP writes files out-of-band), not from `roundGeneratedFiles`, which leaves the files list empty in the review panel. (The collapsed-rail thin icon strip, drag-to-canvas templates, scope-aware submit, and document-template path uniquification all shipped in the Phase 9 UX follow-ups.)

**Next:** (1) reconcile disk-divergence banner + trigger; (2) visual-parity CDP capture at 1280/1440; (3) live MCP probe in the inspector Submit mode.

## What Was Built

**Schema and contract layer (Phase 1).** `lib/canvas/jsoncanvas.ts` gained the v2 type additions — `RelationshipType` (8-value curated catalog), `RELATIONSHIP_TYPES`, `REL_LABELS`, `NodeSource` provenance, `EDGE_ORIGINS`, `SCHEMA_VERSIONS`, and v2 extensions on `EdgeOrigin` (+`'import'`), `NodeMeta` (+`source`, `+template`), `CanvasEdge.meta` (+`rel`), `SessionMeta` (+`baseRevision`, `+pendingReview`), and `FlowcanvasExt.schemaVersion` widened to `'0.1' | '0.2'`. All additions are optional, so every `0.1` board parses unchanged. The pure `lib/canvas/refs.ts` module (`extractRefs`) extracts typed `DocRef[]` from frontmatter `links:` and body markdown links/images. `lib/canvas/brief.ts` was extended to carry groups, typed edges, provenance, and refs through the full brief/merge contract (`BriefNode`, `BriefEdge`, `AgentNode`, `AgentEdge`, `nodeFromAgent`, `buildBrief`, `applyResponse`), and `AGENT_CONTRACT` gained the EXTRACTION / TYPED EDGES / GROUPS rules block.

**Pure library layer (Phase 2).** Three new pure modules: `lib/canvas/review.ts` (`ReviewState`, `ReviewDiff`, `diffDocs` — id-keyed structural diff with order-insensitive canonical compare); `lib/canvas/templates.ts` (`CanvasTemplate`, `TemplateKind`, `instantiateTemplate` — fragment clone with fresh ids, coord rebase, parentId/edge-endpoint remap, provenance stamp); `lib/canvas/edges.ts` extended with `projectLinksForExport` — the `deriveLinkEdges` inverse that projects canvas file→file edges back into per-file `links:` lists for bundle export.

**HTTP API and store integration (Phases 3 + 4).** Four new guarded Node-runtime route handlers: `app/api/canvas/review/route.ts` (GET/POST/DELETE snapshot), `app/api/canvas/active/route.ts` (GET/POST active-board pointer), `app/api/templates/route.ts` (GET list/single), `app/api/canvas/bundle/route.ts` (GET fflate zip with projected `links:` + manifest). The retired `app/api/canvas/links/route.ts` was deleted and `patchLinks` removed from `lib/api.ts`. The Zustand store (`lib/canvas/store.ts`) was rewritten as canvas-authoritative: `load` drops the per-load `reconcileEdges`, runs a one-time `0.1→0.2` migration (bakes `deriveLinkEdges` into persisted edges), and writes the active-board pointer; `onConnect` mints typed `user` edges (`rel:'related'`) with no `links:` write-back; 7 new v2 actions (`submitToAgent`, `reviewDiff`, `acceptRound`, `discardRound`, `navigateRef`, `addTemplate`, `resyncFile`) plus `clearFocus`, `setEdgeRel`, `focusNode`. `lib/canvas/adapter.ts` threads `meta.rel` onto `RFEdge.data` and back.

**MCP sidecar (Phase 5).** `mcp/flowcanvas-mcp.ts` is a thin stdio MCP server (`@modelcontextprotocol/sdk` v1.29.0, `registerTool` + zod) exposing 7 tools that wrap the pure `buildBrief`/`applyResponse` functions and the guarded HTTP API over `FLOWCANVAS_BASE_URL`. All diagnostics go to stderr only; the board is resolved via `.flowcanvas/active-board.json` or an explicit `canvasRef` arg.

**Studio UI shell and surfaces (Phase 6).** Four new React components (`review-panel.tsx`, `template-tray.tsx`, `structure-rail.tsx`, `inspector-rail.tsx`) plus 5 CSS partials (`studio-{shell,review,template,structure,inspector}.css`) imported via `app/globals.css`. The existing `canvas-shell.tsx` was extended into a tri-pane workbench (structure rail · canvas · inspector) with data-attribute–driven collapse; `canvas-toolbar.tsx` gained `toggle-rail-left/right`, `toolbar-submit`, `toolbar-templates`, and `toolbar-bundle`; `labeled-edge.tsx` gained the rel pill + rel picker (`edge-rel-pill`, `edge-rel-picker`, `edge-rel-option`); `frontmatter-view.tsx` and `reader-drawer.tsx` route link-chip and prose-link clicks to `navigateRef`.

## Implementation Details

### Schema / Contract Layer

**File(s):** `lib/canvas/jsoncanvas.ts:15–80`, `lib/canvas/brief.ts:1–30`, `lib/canvas/refs.ts:1–60`

**What it does:** The type root for all v2 features. `RelationshipType` (`jsoncanvas.ts:15`) is an 8-value curated catalog; `RELATIONSHIP_TYPES` (`jsoncanvas.ts:20`) drives the rel picker UI and ships inline in `AGENT_CONTRACT`; `REL_LABELS` (`jsoncanvas.ts:26`) maps each type to its default human label, used by `applyResponse` when the agent omits a label. `NodeSource` (`jsoncanvas.ts:43`) carries `{ path, anchor }` provenance for extracted nodes. `EdgeOrigin` (`jsoncanvas.ts:38`) was widened to include `'import'` (extraction-seeded edges). `SessionMeta` (`jsoncanvas.ts:~65`) carries `baseRevision?` and `pendingReview?` — the review-window state persisted across reload. `FlowcanvasExt.schemaVersion` now accepts `'0.1' | '0.2'`.

**Key decisions:** Every addition is optional — `0.1` boards parse and merge unchanged (verified: all prior `brief.test.ts` seed cases still pass). The `BriefNode.refs` field (`brief.ts:~85`) is populated by `buildBrief` calling `extractRefs` so the agent always sees the node's inbound references. `AGENT_CONTRACT` (`brief.ts:~130`) gained the EXTRACTION / TYPED EDGES / GROUPS block verbatim; the stale "prefer `links:` frontmatter" sentence was removed at review.

---

### Pure Library — `refs.ts`

**File(s):** `lib/canvas/refs.ts:1–60`, `lib/canvas/refs.test.ts`

**What it does:** Pure `extractRefs(basePath, frontmatter, body): DocRef[]`. Extracts three ref kinds — `frontmatter` (`links:` array), `link` (body `[text](x.md)`), and `image` (body `![](img)`). Resolves `./` and `../` relative paths against `basePath`'s directory for body links; treats frontmatter `links:` as root-relative (no resolution against `basePath`). Captures `#anchor`; flags `http(s)` targets as `isExternal`; dedups on `(kind, target, anchor)`. No fs, no DOM.

**Key decisions:** The deviation from the plan's `resolveRel` logic for frontmatter links is intentional — the plan's worked example itself used root-relative frontmatter paths (e.g. `plans/plan.md#phase-1`), and resolving them against `basePath` would produce wrong paths for flowcode-native links. Documented in the Phase 1 log entry and plan spec.

---

### Pure Library — `review.ts`

**File(s):** `lib/canvas/review.ts:1–50`, `lib/canvas/review.test.ts`

**What it does:** `ReviewState` (`review.ts:4`) — the submit-time snapshot persisted to `<board-stem>.review.json`. `ReviewDiff` (`review.ts:12`) — the navigable structural diff. `diffDocs(snapshot, current, roundGeneratedFiles=[])` (`review.ts:~35`) — id-keyed diff (added/updated/removed for nodes + edges, added-only for comments) using order-insensitive canonical stringify (`canon`, `review.ts:20`) so key-reordering doesn't register as a change.

**Key decisions:** The 3rd optional `roundGeneratedFiles` parameter (design showed 2 args) keeps `diffDocs` pure while honoring `ReviewDiff.files == roundGeneratedFiles`; the store passes `ReviewState.roundGeneratedFiles` when it calls `diffDocs`.

---

### Pure Library — `templates.ts`

**File(s):** `lib/canvas/templates.ts:1–45`, `lib/canvas/templates.test.ts`

**What it does:** `CanvasTemplate` (`templates.ts:8`) — a board-fragment schema (`id`, `kind`, `name`, `nodes`, `edges`, `files?`) with coords relative to the fragment's top-left. `instantiateTemplate(t, dropX, dropY, mint)` (`templates.ts:19`) — clones nodes with fresh `n-*` ids via `mint`, rebases coords by `(dropX, dropY)`, remaps `parentId` and edge endpoints through the id map, stamps `meta.origin:'user'` + `meta.template:t.id`, returns `{ nodes, edges, files }`.

---

### Pure Library — `edges.ts` (extended)

**File(s):** `lib/canvas/edges.ts` (extended), `lib/canvas/edges.test.ts`

**What it does:** Added `projectLinksForExport(doc)` — the `deriveLinkEdges` inverse: for each file→file canvas edge, pushes the target path onto the source path's `links:` list, deduplicates, skips non-file endpoints. Consumed by the bundle route to rewrite `links:` in bundled md files. `deriveLinkEdges` and `reconcileEdges` are unchanged and still used for the one-time `0.1→0.2` migration in `store.ts`.

### HTTP API

**File(s):** `app/api/canvas/review/route.ts`, `app/api/canvas/active/route.ts`, `app/api/templates/route.ts`, `app/api/canvas/bundle/route.ts`, `app/api/file/route.ts`, `lib/api.ts`

**What it does:** Four new guarded Node-runtime route handlers, all following the `guardPath`→400/404/500 pattern from `app/api/canvas/route.ts`.

| Route | Methods | Purpose |
|-------|---------|---------|
| `/api/canvas/review` | GET / POST / DELETE | Read/write/clear `<board-stem>.review.json` (Decision 6) |
| `/api/canvas/active` | GET / POST | Read/write `.flowcanvas/active-board.json` pointer (Decision 5) |
| `/api/templates` | GET | List `templates/*.canvas` fragments; `?id=` resolves one |
| `/api/canvas/bundle` | GET | fflate `zipSync` bundle: `.canvas` + projected-`links:` md + assets + manifest |

`app/api/file/route.ts` gained GET (raw read for MCP `read_file`) and DELETE (markdown-only, for `discardRound` rollback). `lib/api.ts` gained `getReview`/`putReview`/`clearReview`, `bundleUrl` (URL helper, not a fetch wrapper — binary download consumed as `<a download href>`), `listTemplates`, `getActive`/`putActive`, `ActiveBoard`, `readFileApi`, `deleteFileApi`; removed `patchLinks`.

**Key decisions:** Bundle dedup: `if (files[rel]) continue` prevents a file referenced by multiple nodes from being added twice. `patchLinks` removal was moved from Phase 3 to Phase 4 to land atomically with the `store.ts` consumer removal (sequencing fix). `bundleUrl` exposed as a URL helper (like `assetUrl`) rather than a `getBundle` fetch wrapper, because binary downloads are consumed as `<a download href>` elements.

---

### Store — Canvas-Authoritative

**File(s):** `lib/canvas/store.ts:1–350`, `lib/canvas/store.test.ts`, `lib/canvas/adapter.ts`

**What it does:** The `load` action is now canvas-authoritative: reads `doc.edges` from disk as truth (no per-load `reconcileEdges`); on `schemaVersion:'0.1'` boards, runs the one-time immutable migration (calls `deriveLinkEdges`, merges any not-yet-persisted `origin:'links'` edges, bumps to `'0.2'`, saves); writes `.flowcanvas/active-board.json`; loads the `ReviewState` from disk when `session.pendingReview` is set.

`onConnect` mints a typed `user` edge (`rel:'related'`, inline editor, no `patchLinks`). `removeEdgeWriteback` removes from `doc.edges` only (no `patchLinks`).

**V2 actions (store.ts:~33–65):**

| Action | Signature | Purpose |
|--------|-----------|---------|
| `submitToAgent` | `(intent: string): Promise<void>` | save → POST review snapshot → POST active-board pointer |
| `reviewDiff` | `(): ReviewDiff \| null` | `diffDocs(snapshot, doc)` when `pendingReview` |
| `acceptRound` | `(): Promise<void>` | DELETE review, clear `pendingReview`, save |
| `discardRound` | `(): Promise<void>` | restore snapshot, delete `roundGeneratedFiles`, DELETE review |
| `navigateRef` | `(sourceNodeId: string, ref: DocRef): Promise<void>` | focus-or-add + `rel:'references'` edge |
| `addTemplate` | `(t: CanvasTemplate, x: number, y: number): Promise<void>` | `instantiateTemplate` + append + dirty |
| `resyncFile` | `(path: string): Promise<void>` | re-read one file's frontmatter from disk; re-derive edges |
| `setEdgeRel` | `(id: string, rel: RelationshipType): void` | typed-edge rel picker |
| `focusNode` | `(id: string): void` | set `focusNodeId` for rail centering |

`addFileNode` signature is `Promise<string>` (returns the new node id so `navigateRef` can immediately draw the edge back).

`lib/canvas/adapter.ts` threads `meta.rel` onto `RFEdge.data` (typed-edge styling by `labeled-edge.tsx`) and back through `toJSONCanvas`.

### MCP Sidecar

**File(s):** `mcp/flowcanvas-mcp.ts:1–60+`, `mcp/README.md`

**What it does:** A thin stdio MCP server (`McpServer` from `@modelcontextprotocol/sdk/server/mcp.js`, `mcp.ts:11`). Configured via `FLOWCANVAS_BASE_URL` (default `http://localhost:3000`, `mcp.ts:23`). Resolves the board via `.flowcanvas/active-board.json` or an explicit `canvasRef` arg (`resolveRef`, `mcp.ts:55`).

**7 tools:**

| Tool | Input | Output | Notes |
|------|-------|--------|-------|
| `get_board` | `{ canvasRef? }` | `DesignBrief` | `buildBrief` over the active/given `.canvas` |
| `apply_response` | `{ canvasRef?, response: AgentResponse }` | `MergeReport` | `applyResponse` + write generated files + write merged `.canvas` |
| `read_file` | `{ path }` | `{ content }` | Reaches `GET /api/file` (Phase-4 addition) |
| `write_file` | `{ path, content }` | `{ ok }` | `.md/.mdx` only; reaches `POST /api/file` |
| `list_dir` | `{ path? }` | `DirEntry[]` | `GET /api/files` |
| `resolve_paths` | `{ paths }` | `ResolvedFile[]` | `POST /api/canvas/resolve` |
| `get_active_board` | — | `ActiveBoard` | `GET /api/canvas/active` |

**Key decisions:** stdio transport — all diagnostics to stderr, stdout owned by the JSON-RPC transport. The sidecar does not modify the in-memory Next.js state; it reads/writes `.canvas` on disk directly; the app reloads on detecting a `session.revision` bump.

**What it does:**

- **Structure rail** (`structure-rail.tsx`): left rail; tree listing of groups + nodes; filter input; click → `focusNode`.
- **Inspector rail** (`inspector-rail.tsx:1–20+`): right inspector; shows IN/OUT typed relations, `meta.source` provenance chips, `DocRef` link chips (→ `navigateRef`); mode-switches to Submit mode (`submitToAgent`) and locks to Review mode when `pendingReview`.
- **Review panel** (`review-panel.tsx:1–20+`): step through `ReviewDiff`; filter by category (`all`/`nodes`/`edges`/`files`/`comments`); prev/next stepper; round-level accept → `acceptRound`, discard → `discardRound`.

- **Typed-edge rel pill/picker** (`labeled-edge.tsx`): `edge-rel-pill` (rel eyebrow + label); `edge-rel-picker` (one `edge-rel-option` per `RELATIONSHIP_TYPES` entry + `edge-label-input`) → `setEdgeRel`/`relabelEdge`. `import` stroke added to the exhaustive `Record<EdgeOrigin, string>` (violet) as the Phase-1 ripple fix.
- **Reference navigation** (`frontmatter-view.tsx`, `reader-drawer.tsx`, `nodes/markdown-node.tsx`): `↗` link-chip is a clickable button calling `navigateRef(sourceNodeId, ref)` when `sourceNodeId` is supplied; reader prose delegated `a[href]` click handler intercepts `.md`/asset targets.
- **Toolbar** (`canvas-toolbar.tsx`): `toggle-rail-left`/`-right`, `toolbar-submit` (opens inspector Submit mode), `toolbar-templates` (opens template tray), `toolbar-bundle` (`<a download>` via `bundleUrl`).

## Deviations from Plan

| Plan Spec | Actual Implementation | Reason |
|-----------|-----------------------|--------|
| `diffDocs(snapshot, current)` — 2 args | `diffDocs(snapshot, current, roundGeneratedFiles = [])` — 3rd optional arg | Keeps `diffDocs` pure while honoring `ReviewDiff.files == roundGeneratedFiles`; the store passes `ReviewState.roundGeneratedFiles` |
| `addFileNode(path, x, y): void` | `addFileNode(path, x, y): Promise<string>` — returns new node id | `navigateRef` needs the id immediately to draw the edge back to the source |
| `reviewDiff().files` from stored `roundGeneratedFiles` | Derived from added file-node paths | MCP sidecar writes generated files out-of-band (not via store); `roundGeneratedFiles` stays `[]` so `files` is derived from the diff's added file nodes |
| Submit lives in `export-panel` as a tab | Submit lives in the inspector's Submit mode | Matches mockup `inspector-submit`; avoids a duplicate surface; `export-panel` stays the clipboard Export/Import fallback |
| `getBundle(path)` fetch wrapper in `lib/api.ts` | `bundleUrl(path)` URL helper | Binary download is consumed as an `<a download href>`; a fetch wrapper would require blob-piping with no benefit |
| `/api/canvas/links` route + `patchLinks` deleted in Phase 3 | Deleted in Phase 4 | Sequencing fix: `store.ts:138,161` + `store.test.ts` still consumed `patchLinks` in Phase 3; removal must be atomic with the `onConnect`/`removeEdgeWriteback` change |
| `get_board` stamps `session.lastBriefId` | Stamped (Phase 7, D3) via a `bump:false` POST | Phase 5 shipped without the stamp (`apply_response` always `stale:true`); Phase 7 added it so the concurrency check is functional without churning the revision |
| `extractRefs` frontmatter `links:` resolved against `basePath` | Treated as root-relative | The plan's own worked example uses root-relative frontmatter paths; resolving them against `basePath` produces wrong paths for flowcode-native links |
