Commerce Platform — System Design

A storefront and order pipeline decomposed into edge, core services, and a data/event tier. This document is the source-of-truth for content; the canvas on the right is the relation graph extracted over it.

Every component below maps to exactly one board node. The agent extracted them and stamped each with meta.source provenance back to the heading you are reading.

Overview

Traffic enters through the edge tier, fans out to core services, and persists through the data & events tier. Two external systems are integrated at the boundary.

Edge & BFF

The terminates TLS and routes POST /orders to the order service; the shapes responses for the storefront.

Core Services

Auth Service

The issues sessions; the gateway depends-on it. It also informs the of sign-in events, and exposes as a shared contract. See the off-board for backoff rules.

Order Service

The owns the order lifecycle: it calls charge on payments, produces the order.created event, depends-on the orders database, and implements the order-schema.md contract.

  • Idempotent create on POST /orders (client-supplied key).
  • Emits order.created after a successful charge — never before.
  • Reads inventory hints from the inventory service (non-blocking).

Payment Service

The wraps the gateway and retries per the policy above.

Inventory Service

The tracks stock and informs order placement; the edge is links:-derived (file→file), so it draws dashed.

Data & Events

The is the system-of-record; the fronts it for hot reads. The carries order.created to downstream consumers.

External Systems

Two boundary integrations are linked, not owned: for charges and for receipts.

Contracts

The order schema is the shared contract between the order service and its consumers:

# order-schema.md (excerpt)
type Order = {
  id: string,        // idempotency key
  items: LineItem[],
  total: number,      // minor units
  status: "created" | "charged" | "failed"
}
