# Technical Overview — 006-semantic-edges Semantic Edges & Connection Ports

- Reworked the canvas edge system into three operator-locked specs — dot-anchored connection ports, a reusable color picker, and flow-typed edges whose color/line/head encode meaning — plus Shift-snap line angles, all with full human↔agent parity.
- Status complete; dated 2026-06-30; generated by the post-execution pipeline (code-verified against the working tree + live CDP).
- Role: authoritative source for `006-semantic-edges-changelog.md` and `006-semantic-edges-test-notes.md`; reconciled against code by the Code Explorer audit.

## Summary

**What:** Edges now anchor to real, reusable, movable **connection dots** (ports), carry a **semantic flow type** that drives their color/line/arrowhead via a legend that doubles as a type picker, share a reusable **`<ColorPicker>`** (edges + node text/fill), and snap to 45° angles while bending the line with Shift held.

**Why:** The 005-edges work regressed the connection model (floating endpoints, hidden dots, loose arrowheads) and left edges meaningless (style decoupled from relationship) and color unmanaged. A system-design canvas must be read at a glance by color/line/head and must let connections land on real anchors.

**Risks:** Connection geometry now flows through ports (a load-time normalize step); a board opened in the app migrates `0.4 → 0.5` and persists (on-disk fixtures bump on open — `examples/commerce-platform.canvas` was adopted as the `0.5` showcase). `meta.rel` is retained one version (legacy) — its full retirement is a future cleanup.

**Tech-Debt:** Per-phase module-doc sync for Phases 3–4 was consolidated into this pipeline; `fc-relpick*`/`fc-edge-label__rel` class names are kept as stable handles across the rel→type rename (documented). The auto/body-drop dedup uses a slot tolerance distinct from the migration's reuse tolerance (named separately).

**Next:** Retire `meta.rel` from the wire once a version has passed; optional `EdgeLegend` single-`useStore` micro-optimization; consider agent-authored port ids if cross-board merge ever needs them.

## What Was Built

The edge endpoint stopped being a transient floating point and became a **connection port** — a stable dot on the node perimeter (`ConnectionPort {id, side, t}` in `node.meta.ports`) that an edge references by id (`fromPort`/`toPort`). Ports are the single geometric source of truth: a `0.4 → 0.5` migration seeds a port for every existing edge endpoint and an idempotent `normalizePorts` heals hand/agent edges at load, so the renderer only ever reads ports (`labeled-edge` resolves the endpoint via `portPoint({side,t})` against the live node rect, so the arrowhead seats exactly in the dot and follows node/port moves with no React-Flow re-measure lag). Connecting reuses the dot dropped on or creates one (a side "add" handle spreads dots along a side; a body drop reuses a near dot or auto-places one); Alt-dragging a dot slides it and every anchored edge follows.

Meaning comes from an `EdgeType` flow taxonomy (`data-flow · request · response · event · dependency · reference`) bound in one `EDGE_TYPE_STYLE` constant to a default `{color, line, fromEnd, toEnd}`. An edge's rendered style resolves `per-edge override → EDGE_TYPE_STYLE[edgeType] → provenance`. An on-canvas `<EdgeLegend>` lists the six types and **doubles as the type picker**: select an edge, click a legend row, the edge restyles. A reusable `<ColorPicker>` (preset chips + native picker + clear) was extracted and wired into both the edge style panel (per-edge color override) and `node-format-bar` (text + fill). Holding Shift while bending a line snaps each segment to 45° (`snapAngle`).

Full agent parity closes the loop: `AgentEdge`/`BriefEdge` gain `edgeType`; `buildBrief` emits it and `applyResponse` threads it (create defaults `'reference'`, update preserves); the generation-kit `schemaContract` documents the taxonomy + legend as the primary edge meaning (with the mirror `docs/flowcanvas-agent-contract.md` regenerated from it); the MCP `apply_response` carries `edgeType` unchanged via its `z.any()` edge schema. Schema `0.5` is the new persisted version (boards upgrade on open).

## Implementation Details

### Schema + pure geometry libs

**File(s):** `lib/canvas/jsoncanvas.ts`, `lib/canvas/ports.ts` (new), `lib/canvas/edge-geometry.ts`, `lib/canvas/migrate.ts`, `lib/canvas/validate.ts`
**What it does:** `jsoncanvas.ts` adds `ConnectionPort`, the `EdgeType`/`EDGE_TYPES`/`EdgeTypeStyle`/`EDGE_TYPE_STYLE` legend, `REL_TO_EDGE_TYPE`, `NodeMeta.ports`, `CanvasEdge.fromPort`/`toPort`/`meta.edgeType`, and schema `0.5`. New pure `ports.ts` = `portPoint`/`sideAndT`/`portAt`/`autoPort`. `edge-geometry.ts` gains `snapAngle`. `migrate.ts` adds the `0.4 → 0.5` step (seed ports via `seedEdgePorts`, map `rel → edgeType`) + the exported idempotent `normalizePorts`. `validate.ts` widens the zod enum; new fields ride `.passthrough()`.
**Key decisions:** ports materialize at load and render reads only (design D4) — keeps `labeled-edge`/`adapter` side-effect-free; `ports.ts` stays DOM-free (pure-lib boundary, unit-tested).

### Adapter + store (rendering + interaction)

**File(s):** `lib/canvas/adapter.ts`, `lib/canvas/store.ts`
**What it does:** the adapter builds a `portIndex` and resolves `fromPort`/`toPort` → `{side,t}` into edge `data.fromPortST`/`toPortST`; the store adds `addPort`/`movePort`/`setEdgeType`, rewrites `onConnect` (reuse-or-create via `portForConnect`/`firstFreeT`), runs `normalizePorts` in `load`, and births new boards at `0.5`. `setEdgeType` applies a type's legend default by clearing the per-edge overrides it supersedes.
**Key decisions:** resolving `{side,t}` in the adapter (not relying on RF handle re-measurement) makes port/node moves lag-free; `setEdgeType` clears `color/line/fromEnd/toEnd` so the edge immediately reads as its type, with the picker as the escape hatch.

### Ports + legend + color-picker UI

**File(s):** `components/canvas/nodes/port-handles.tsx` (new), `node-frame.tsx` + leaf/group/fallback nodes, `components/canvas/edges/labeled-edge.tsx`, `components/canvas/use-canvas-handlers.ts`, `components/canvas/legend.tsx` (new), `components/ui/color-picker.tsx` (new), `components/canvas/nodes/node-format-bar.tsx`, `app/styles/{edges,studio-edge}.css`
**What it does:** `<PortHandles>` renders one always-visible dot per port (+ 4 faint side "add" handles); `labeled-edge` anchors endpoints at `portPoint`, resolves style from `edgeType`, swaps the rel pill/picker for an edgeType pill/`EdgeTypePicker`, and uses `<ColorPicker>` in the style panel; a single window CAPTURE-phase listener in `use-canvas-handlers` drives Alt-drag-to-move; `<EdgeLegend>` doubles as the type picker; `<ColorPicker>` is shared by edges + node-format-bar.
**Key decisions:** Alt-drag handled at the canvas level (RF Handle doesn't reliably forward `onPointerDown`); the legend `stopPropagation`s pointerdown (else the edge deselects before the click) and is pinned top-left (bottom-left collides with RF Controls / falls below the fold — both CDP-diagnosed).

### Agent parity

**File(s):** `lib/canvas/brief.ts`, `lib/canvas/generation-kit.ts`, `docs/flowcanvas-agent-contract.md`, `mcp/flowcanvas-mcp.ts`
**What it does:** `edgeType` added to `AgentEdge`/`BriefEdge`, emitted by `buildBrief`, threaded by `applyResponse`; the generation-kit contract documents the taxonomy + legend; the mirror doc is regenerated from `kitSections().schemaContract`; the MCP fresh-board bumped to `0.5` (edges pass `edgeType` via `z.any()`).
**Key decisions:** agents author edges by `edgeType` + optional sides; ports seed at load — agents don't mint port ids (app-managed geometry).
