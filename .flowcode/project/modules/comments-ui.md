# Comments UI

- Owns the two React components that render the comment system on the canvas: the `CommentLayer` pin overlay (projects anchors to screen coordinates, handles comment-mode click-to-drop) and the `CommentThread` glass popover (flat thread, reply composer, resolve toggle). Everything that knows about the DOM or React flow coordinates lives here; anchor geometry math lives in the sibling pure module `comments` (`lib/canvas/comments.ts`).
- Path: `components/canvas/comment-layer.tsx` + `components/canvas/comment-thread.tsx`; stack: TypeScript 5 / React 19 / `@xyflow/react` ^12; styles in `app/styles/comments.css`.
- Public API: `CommentLayer` (zero-prop overlay) · `CommentThread` (memo'd tethered popover, driven by `CommentThreadProps`).

## Purpose

`CommentLayer` is a fixed-position React overlay (z-index 6) mounted once inside the canvas pane by `canvas-shell.tsx`. It subscribes to `useNodes()` and `useViewport()` so it re-renders whenever nodes are dragged, resized, measured, or the viewport pans/zooms — this makes every pin track live node geometry without a separate effect. In `comment` mode it captures surface clicks, converts them to flow-space coordinates, calls `anchorForPoint` from the pure comments library to determine node vs. canvas anchoring, and stores a draft anchor in local state. When the user types and submits a draft, it calls `addComment` on the Zustand store. Existing thread roots are rendered as teardrop `.fc-pin` buttons; clicking one opens a `CommentThread`.

`CommentThread` is a `memo()`'d glass popover tethered to its pin via a CSS triangle beak. It doubles as the draft composer (when `root` is `null`) and the open-thread view (flat root + replies, reply box, resolve button). The parent (`CommentLayer`) pre-computes the popover's absolute position — including the left/right flip and vertical clamp — and passes it down as `pos`, `side`, and `beakTop`.

Comment mode is entered via the toolbar's `toolbar-comment-mode` button, which calls `store.setMode('comment')`.

## Key Insights

**Live-measured geometry projection — why `useNodes()` + `useViewport()` are both subscribed:**

`CommentLayer` calls `useViewport()` with no destructuring (`comment-layer.tsx:22`) solely for its subscription side-effect: it forces a re-render on every pan and zoom so `flowToScreenPosition` recomputes pin coordinates against the current viewport transform. `useNodes()` (`comment-layer.tsx:23`) forces a re-render on every drag and measure event and supplies the node list that builds the `geom` array. Together they make pins track all interactive state continuously without any imperative listener.

**Absolute position for grouped children:**

For grouped child nodes, `node.position` from `useNodes()` is relative to the parent. To place a pin correctly over a child node, `CommentLayer` reads `getInternalNode(n.id)?.internals.positionAbsolute` and falls back to `n.position` only when the internal node is unavailable (`comment-layer.tsx:47–53`). Without this, pins on grouped-child nodes would be offset by the parent's position.

**Pointer-events overlay pattern:**

`.fc-comment-layer` has `pointer-events: none` by default so no canvas interaction is blocked. The CSS rule `.fc-comment-layer[data-mode]` switches to `pointer-events: auto; cursor: crosshair` (`app/styles/comments.css:7-8`). The React side sets `data-mode={commentMode || undefined}` — `undefined` removes the attribute, reverting to click-through (`comment-layer.tsx:115`). Pins and popovers keep their own `pointer-events: auto` regardless of mode.

**Draft-mode render-phase reset (not useEffect):**

When comment mode turns off, the in-progress draft anchor is cleared using a render-phase state comparison pattern (`comment-layer.tsx:36–40`) — the React-recommended alternative to `setState` in `useEffect` for derived-state resets. This avoids the extra render cycle that an effect would introduce.

**Tethered-beak popover placement math:**

`placePopover` (`comment-layer.tsx:70–83`) places the 320px-wide popover with a 14px gap from the pin, then flips to the left side when `aimX + GAP + W > vw - 12`. The vertical position is clamped to `[12, vh - 300]` so the popover never overflows the viewport top or bottom. `beakTop` = `aimY - top` aligns the CSS beak triangle to the pin badge centre (the badge renders `translate(-50%, -100%)`, so badge centre is at `pt.y - 13` — `comment-layer.tsx:76`).

**`CommentThread` is stateless-by-design for placement:**

`CommentThread` owns no position state of its own; all placement is computed by `CommentLayer` and passed down as `pos`, `side`, `beakTop`. This separation lets the parent own all geometry logic while the popover stays a pure controlled render. `CommentThread` is `memo()`'d to avoid re-renders when unrelated store slices change.

**`LOCAL_AUTHOR` is hardcoded:**

`comment-layer.tsx:10` defines `const LOCAL_AUTHOR = 'human:you'`. This is the Phase 6 placeholder. Phase 7 agent replies arrive via the brief round-trip (`replyComment` called with `"agent:<model>"`), not via this component. There is no configuration or auth plumbing for the human author string — changing it requires editing the source file.

**`resolveComment` is a true no-op for invalid calls:**

The store implementation (`store.ts:451–460`) checks that `rootId` refers to an existing root (not a reply) that is not already resolved. A stale call from a cached closure never spuriously marks `dirty: true`.

**Gotchas & invariants:**

- `CommentLayer` returns `null` when `doc` is `null` — if the store has not finished loading the board, no overlay is rendered and no error is thrown (`comment-layer.tsx:107`).
- `project(anchor)` returns `null` when the anchor references a node that has been deleted; the root's `.map()` silently skips it (`comment-layer.tsx:120`). Stale comment anchors are never surfaced as errors.
- The Esc key handler (`comment-layer.tsx:97–105`) has a two-step behaviour: first Esc closes the open thread or draft without leaving comment mode; a second Esc exits comment mode. Agents modifying keyboard UX must preserve this two-step contract.
- `CommentThread`'s `onClick` stops propagation (`comment-thread.tsx:77`) so clicks inside the popover never fall through to `CommentLayer`'s `onSurfaceClick` and accidentally drop a new pin.
- Agent replies land in the flat `replies` array under the root; there is no nesting — the data model (`Comment.parentId`) supports exactly one level of threading.

## Known Gaps

- `LOCAL_AUTHOR` (`comment-layer.tsx:10`) is a hardcoded placeholder (`'human:you'`); there is no user identity or auth plumbing — a future identity system would need to inject the current user's author string.
- No vitest unit coverage for `CommentLayer` or `CommentThread`; the only automated coverage is the headless-Chrome `smoke:render` smoke.
- Drag-while-open is not debounced: during a fast drag the popover re-positions on every frame via `placePopover`. This is acceptable for a 320px popover but could become janky on a slow device.
