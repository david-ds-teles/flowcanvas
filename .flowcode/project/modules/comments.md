# Comments (Anchor Geometry)

- Pure TypeScript math module: converts a flow-space click into a `CommentAnchor` (node-relative or canvas-absolute) and projects anchors back to flow points for pin rendering.

## Purpose

`lib/canvas/comments.ts` owns the two pure geometric primitives that power the comment-pinning system. `anchorForPoint` accepts a flow-space click coordinate and a list of live measured node boxes, and resolves it to either a node-relative anchor (0..1 fractions of the node's measured width/height) or a bare canvas coordinate anchor. `anchorToFlowPoint` is the inverse: given an anchor and the current node-geometry map it returns the absolute flow-space point the pin should render at, following the node as it is dragged or resized, or `null` when the anchored node has been removed from the board. The module is intentionally DOM-free and React-free so it can be exercised under Vitest without a browser context. The sole consumer is the React overlay `components/canvas/comment-layer.tsx` (module `comments-ui`), which supplies the live `useNodes` + `useViewport` geometry and feeds the results to `flowToScreenPosition` for screen placement.

## Usage Examples

```ts
// Derived from lib/canvas/comments.test.ts:10-56

import { anchorForPoint, anchorToFlowPoint, type NodeGeom } from '@/lib/canvas/comments'

const nodes: NodeGeom[] = [
  { id: 'card-1', x: 0,   y: 0,  width: 100, height: 80 },
  { id: 'card-2', x: 200, y: 50, width: 100, height: 100 },
]

// ‚îÄ‚îÄ Click inside card-1 at its centre ‚îÄ‚îÄ
const anchor = anchorForPoint({ x: 50, y: 40 }, nodes)
// => { kind: 'node', nodeId: 'card-1', offsetX: 0.5, offsetY: 0.5 }

// ‚îÄ‚îÄ Click on bare canvas ‚îÄ‚îÄ
const canvasAnchor = anchorForPoint({ x: 500, y: 500 }, nodes)
// => { kind: 'canvas', x: 500, y: 500 }

// ‚îÄ‚îÄ Project the node anchor back to a flow point ‚îÄ‚îÄ
const byId = new Map(nodes.map((n) => [n.id, n]))
const flowPoint = anchorToFlowPoint(anchor, byId)
// => { x: 50, y: 40 }  ‚Äî round-trip identity for interior points

// ‚îÄ‚îÄ Node removed from board ‚îÄ‚îÄ
const gone = anchorToFlowPoint({ kind: 'node', nodeId: 'deleted', offsetX: 0.5, offsetY: 0.5 }, byId)
// => null
```

## Key Insights

**Conventions & patterns:**

- Follows the project's strict `lib/canvas/*` = pure TypeScript rule: no DOM, no React, no `@xyflow/react` imports at runtime ‚Äî only a type-import from `jsoncanvas.ts`. This boundary is what allows Vitest to run it without jsdom (`lib/canvas/comments.ts:1`).
- Hit-testing traverses `nodes` back-to-front (`for i = nodes.length-1 downto 0`) so the last-drawn (topmost) node wins when boxes overlap (`lib/canvas/comments.ts:30`). This mirrors the painter's-algorithm z-order used by React Flow.
- Zero-sized boxes are explicitly skipped (`width <= 0 || height <= 0`) so a not-yet-measured node (React Flow measures asynchronously after first render) cannot accidentally capture clicks (`lib/canvas/comments.ts:32`).
- `anchorToFlowPoint` returns `null` (not `undefined`, not a throw) when the referenced node is gone ‚Äî the caller (`comment-layer.tsx`) must filter nulls before rendering pins. This is an explicit API contract, not an oversight.
- Offsets are clamped to `[0, 1]` with the inclusive-boundary semantics that an exact corner click (`x === n.x + n.width`) returns `1.0`, not a value slightly above 1 that would escape the node box (`lib/canvas/comments.ts:21, 38`).

**Gotchas & invariants:**

- The `nodes: NodeGeom[]` array passed to `anchorForPoint` must be in paint order (bottom-to-top) for the back-to-front traversal to pick the correct winner on overlap. React Flow's `useNodes()` returns nodes in this order, but a caller building the array from another source must sort accordingly.
- `offsetX` and `offsetY` are fractions of the **measured** node box (`width`/`height` from `NodeGeom`), not the authored doc box stored in the `.canvas` file. When `anchorToFlowPoint` is called, the same measured geometry map must be supplied ‚Äî mixing authored and measured dimensions would misplace pins.
- The module was introduced in plan `001` phase 6-7 (commit `f3a92ff`) and has not been touched since; it predates the v2 `RelationshipType` and `NodeSource` additions and is unaffected by them.
- No test for `anchorForPoint` with an empty `nodes` array ‚Äî but the function body handles it correctly (the loop never executes, the canvas-anchor fallback fires). Not a gap requiring a test, just worth knowing.
