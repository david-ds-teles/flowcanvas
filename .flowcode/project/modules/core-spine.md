## Purpose

`core-spine.tsx` is the docked living core-markdown spine introduced in plan 004 Phase 4. Unlike the overlay `reader-drawer.tsx` (read-only, transient), this pane is a permanent flex column mounted between the canvas and the inspector in `canvas-shell.tsx`. It owns four coupled concerns: (1) **rendering** — fetches `session.coreDocPath` via `GET /api/render` and injects full-fidelity shiki HTML via `dangerouslySetInnerHTML` (safe: the render route passes output through `rehype-sanitize`); (2) **editing** — a textarea over the raw markdown draft with a dirty-flag header badge, blocked-submit guard while a review round is open (Decision 3 single-open-round invariant); (3) **doc switching** — a `<select>` switcher over `citedDocPaths(nodes)` that rebinds the spine to any cited doc via `setCoreDoc` (Q4 operator override); and (4) **bidirectional pulse** — a section outline with per-heading component-count badges that drives `highlightComponents` (spine → canvas), and a `spineHighlightAnchor` subscription that scrolls + one-shot pulses the matching heading element (canvas → spine). Prose relative links that resolve to an on-board node call `focusNode` rather than navigating; no phantom-source edge is created.

## Key Insights

**Conventions & patterns:**

- *`'use client'` component.* `CoreSpine` is a Next.js App Router client component — it uses React state, effects, and browser APIs (`CSS.escape`, `scrollIntoView`, `classList`). It must never be imported by a server component or a server route handler.
- *Thin shell over Zustand.* All persistent state lives in the store (`coreDocBody`, `coreDocDraft`, `coreDocDirty`, `spineHighlightAnchor`). The component holds only the three pieces of transient UI state it alone owns: `editing` (textarea toggle), `submitting` (async lock), `err` (submit error message), and `result` (the fetched HTML keyed by `forPath`).
- *`forPath` keyed render result.* The local `result` state (`{ forPath, html, error }`) guards against races: `ready` is true only when `result.forPath === coreDocPath` (`core-spine.tsx:47`). Switching to a different doc immediately shows "Rendering…" until the new fetch lands, preventing the previous doc's HTML from flashing under the new doc's outline.
- *`live` fetch guard.* Every `useEffect` fetch sets `let live = true` and returns a cleanup that sets `live = false` (`core-spine.tsx:50-59`). State is only updated when `live` is still true, preventing React "can't update state on unmounted component" warnings and stale-fetch races on rapid path changes.
- *One-shot CSS pulse (no persistent style).* The `is-pulse` class is added to the heading element via `classList.add('is-pulse')` and removed after 1500 ms via `setTimeout` (`core-spine.tsx:68-71`). The effect cleanup also removes it immediately on unmount or re-fire (`core-spine.tsx:72`). The `@keyframes fc-spine-pulse` animation fades to nothing, so the class can linger harmlessly between cleanup and the next `setTimeout` tick — there is no visible artifact. The canvas-side `.fc-rf--linked` pulse uses the same design: a one-shot keyframe that decays to `box-shadow: none` (`studio-spine.css:134-137`).
- *`CSS.escape()` is mandatory.* Heading anchors produced by `github-slugger` can contain characters that are valid in HTML `id` attributes but invalid as CSS selectors (e.g. colons, dots). Both `querySelector` calls (`core-spine.tsx:65, 77`) must use `CSS.escape(anchor)` or they silently return `null` for any anchor containing a special character.
- *Switcher visibility rule.* The switcher renders when `(cited.length > 1 || (!coreDocPath && cited.length >= 1))` (`core-spine.tsx:132`). The asymmetry is deliberate: once a doc is bound (coreDocPath set), a one-cited-doc board hides the switcher to reduce chrome. An unbound board with exactly one candidate shows the switcher so the user can bind it.
- *Prose relative-link → `focusNode`, not navigation.* The delegated click handler (`core-spine.tsx:83-92`) intercepts only href values that are neither in-doc anchors (`#`) nor absolute URIs (`^[a-z]+:`). It resolves the target using `normPath` against every `isFileNode` on the board. If found, `focusNode(id)` is called; if not found, the click is suppressed (no external navigation, no phantom-source edge). This is correct: the core doc is not itself a board node, so its links have no board-side source to draw an edge from.

**Gotchas & invariants:**

- **Single-open-round invariant (Decision 3).** `submitCoreDocEdit` throws `'A review round is already open…'` synchronously when `session.pendingReview` is true (`store.ts:723-725`). The submit button is also `disabled` (`core-spine.tsx:200`) and `aria-disabled` at the same condition. The footer also renders a plain-language note (`core-spine.tsx:195`). All three guards must stay in sync — removing any one creates a path to opening a second review round, which corrupts the single `snapshot` / `pendingReview` slot.
- **`dangerouslySetInnerHTML` is safe here.** The HTML fed to `fc-spine__prose` comes exclusively from `GET /api/render`, which runs the content through `rehype-sanitize` before returning it (`lib/render-md.ts`). Never feed user-typed `coreDocDraft` directly into `dangerouslySetInnerHTML` — that would be an XSS vector.
- **Heading anchor parity is a hard cross-module invariant.** `outlineOf` produces anchor slugs via `github-slugger`. The same slugger underpins `rehype-slug` in the render pipeline (so headings in the rendered HTML carry matching `id` attributes) and the agent's `meta.source.anchor` convention. If `render-md.ts` changes the pipeline ordering so `rehype-slug` runs after `rehype-sanitize`, the sanitizer may strip the `id` attributes and all `querySelector` calls in `CoreSpine` will silently return `null`. See `spine.md § Key Insights` for the full parity discussion.
- **`buildSourceIndex` is called inside `useMemo` on `[doc, coreDocPath]`.** It re-runs on every doc identity change, not just when nodes change. Because Zustand produces a new `doc` reference on each `save`, the index is rebuilt frequently. This is intentional (correctness over micro-optimisation) but a very large board (hundreds of nodes) may show memoisation pressure here first.
- **Edit mode hides the outline.** The condition `!editing && outline.length > 0` (`core-spine.tsx:148`) intentionally hides the section outline while the textarea is open. The outline links invoke `highlightComponents` which scrolls the prose; that scroll would fight with the user's cursor position in the textarea. The outline reappears when the user clicks "Done".
- **`focusNode` does not open the spine on a fresh board.** `focusNode` in the store only sets `focusNodeId` for `FocusBridge` to center the viewport; it does not set `session.coreDocPath`. If the board has no bound core doc, `spineAvailable` in canvas-shell evaluates to false and the pane is not mounted regardless.
