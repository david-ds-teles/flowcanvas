# Frontmatter

- Owns `gray-matter` parse/stringify for every `.md` file touched by server-side routes — the single ingestion point for raw markdown content across all Route Handlers.
- Path: `lib/canvas/frontmatter.ts`; stack: TypeScript 5, server-side only (`gray-matter`).
- Public API: `parseFile(raw)` → `{frontmatter, body, truncated}`; `stringifyFile(frontmatter, body)` → `string`; constant `BODY_CAP = 40_000`.
- Consumed by `canvas/resolve/route.ts` (batch frontmatter hydration for the canvas store), `canvas/bundle/route.ts` (projected-`links:` re-serialization into the zip), and `render/route.ts` (body extraction for the shiki pipeline).
- Status active; generated by bootstrap; last updated 2026-06-29.

---

## Purpose

`frontmatter.ts` is a thin server-side wrapper over `gray-matter` with one added policy: bodies longer than `BODY_CAP` (40 000 chars) are hard-truncated at that boundary and the `truncated: true` flag is returned alongside the cut body. `parseFile` is the entry point for reading any `.md` file — it strips YAML frontmatter, applies the cap, and returns three fields. `stringifyFile` is the exit point — it reassembles updated frontmatter back onto a body using `matter.stringify` (js-yaml under the hood) for round-trips where frontmatter fields change (e.g. the bundle route injecting projected `links:`). Every Route Handler that touches raw markdown goes through one of these two functions; none of them re-implement frontmatter parsing directly.

### Exceptions / Errors

The module performs no error handling. `gray-matter` will throw a `YAMLException` on malformed YAML front matter; `matter.stringify` will throw if `js-yaml` cannot serialize a value. Both propagate unhandled to the calling Route Handler, which maps them to a 500 response via its own catch block.

| Name | Raised When | Caught By |
|------|-------------|-----------|
| `YAMLException` (gray-matter / js-yaml) | Malformed YAML frontmatter block in `parseFile` | Caller's `catch (e)` → 500 |
| Serialization error (js-yaml) | Unserializable value type passed to `stringifyFile` | Caller's `catch (e)` → 500 |

// entry.truncated   → false

## Key Insights

**Conventions & patterns:**

- Pure module: no DOM, no React, no filesystem I/O. Accepts a raw string, returns a typed plain object. Sits strictly in the `lib/canvas/*` pure-library layer (`project-overview.md § Code Style`).
- Functional style with no state: both exports are stateless functions plus one constant. No class, no singleton, no side effects.
- Error propagation: all exceptions bubble to the caller. The module intentionally contains no `try/catch` — the Route Handler owns the error boundary.

**Gotchas & invariants:**

- `BODY_CAP` is a hard slice at 40 000 characters, not a sentence or word boundary. A body of 40 001 chars returns `body.slice(0, 40000)` with no trailing ellipsis. Callers (and the agent via `brief.ts`'s `truncated?: boolean` field) must check `truncated: true` to know the body was cut.
- `parseFile` returns the `content` field from `matter(raw)` — this is the body *with the frontmatter block already stripped*. Callers never see the raw `---` delimiters in `body`.
- `stringifyFile` delegates to `matter.stringify(body, frontmatter)` which uses `js-yaml` for YAML serialization. Scalar values may be re-quoted in a different style from the original file (e.g. an unquoted `yes` may become `'yes'`, a double-quoted string may become single-quoted). The data values are preserved faithfully; only the YAML quoting style may change. This is cosmetic for most fields but can cause noisy diffs if the file is also edited by hand.
- The bundle route (`canvas/bundle/route.ts:42`) reads raw bytes with `matter()` directly (not via `parseFile`) because it needs the full-length `content` without the 40 000-char cap, then calls `stringifyFile` for the output. This is intentional: `BODY_CAP` applies to agent-facing resolved bodies, not to the portable bundle.
- `gray-matter` is server-side only. It is not imported on the client — the `lib/canvas/frontmatter.ts` module must never be added to a client component import chain.
- `BODY_CAP` is referenced by name in `lib/canvas/brief.ts` comments (line 46) to document the `truncated` field on `BriefNode`. If `BODY_CAP` is raised or removed, update `brief.ts`'s inline documentation accordingly.
