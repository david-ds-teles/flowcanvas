## Purpose

`fs-guard` is the sole filesystem security boundary for the Flowcanvas Node runtime. It resolves `FLOWCANVAS_ROOT` (defaulting to `process.cwd()`) once at module-load time into the module-level constant `ROOT`, and exports `guardPath(rel)` — the function every route handler calls before touching `fs`. `guardPath` resolves `rel` against `ROOT` using Node `path.resolve` (lexical normalization: collapses `../` traversals and absolute paths) and throws `GuardError` if the resulting absolute path is not `ROOT` itself or a strict descendant (`startsWith(ROOT + sep)`). All eleven route handlers import and call it; the `GuardError` class is the agreed contract that callers map to HTTP 400. The module has no runtime dependencies beyond Node's built-in `node:path` — no framework, no I/O.

## Usage Examples

The canonical call-site pattern, as it appears in every route handler:

```typescript
// Real call site: app/api/canvas/route.ts:13-16
import { guardPath, GuardError } from '@/lib/fs-guard'
import { readFile } from 'node:fs/promises'

// Inside a Next.js route handler (Node runtime):
try {
  const abs = guardPath(rel)           // rel = query param, e.g. 'examples/flowcanvas.canvas'
  const doc = JSON.parse(await readFile(abs, 'utf8'))
  return NextResponse.json({ doc })
} catch (e) {
  if (e instanceof GuardError) return NextResponse.json({ error: e.message }, { status: 400 })
  if ((e as NodeJS.ErrnoException).code === 'ENOENT') return NextResponse.json({ error: 'not found' }, { status: 404 })
  return NextResponse.json({ error: String(e) }, { status: 500 })
}
```

`app/api/canvas/route.ts:13-16` — demonstrates the `guardPath` → `readFile` → `GuardError`→400 / `ENOENT`→404 / else→500 error cascade that all eleven route handlers follow. The path-escape case (`../../etc/passwd`) is exercised by the vitest route-contract test at `app/api/routes-contract.test.ts:68-69`.

## Key Insights

**Conventions & patterns:**

- **Module-load-time singleton.** `ROOT` is resolved exactly once when Node first `require`/`import`s the module (`lib/fs-guard.ts:7`). This means `FLOWCANVAS_ROOT` must be set in the environment *before* the process starts — changing it at runtime has no effect. All eleven route handlers share the same `ROOT` value.
- **Strict descent check.** The guard uses `abs.startsWith(ROOT + path.sep)` (`lib/fs-guard.ts:12`), not just `startsWith(ROOT)`. This prevents a sibling directory whose name starts with the root directory name (e.g. `ROOT=/data/project` and `abs=/data/project-evil/file`) from passing. The `abs !== ROOT` check also blocks the root itself when `rel` resolves to it exactly (callers pass file paths, not the root directory itself).
- **Guard-first route pattern.** Project convention (documented in `project-overview.md § Code Style & Conventions`): `guardPath(rel)` is always the first fs operation inside a `try` block; `GuardError`→400 is always the first `catch` branch. This pattern is uniform across all eleven consumers.
- **Error message reveals the bad `rel`.** `GuardError` is constructed with `\`path escapes root: ${rel}\`` (`lib/fs-guard.ts:12`). Callers forward `e.message` to the HTTP response body (`{ error: e.message }`), so the escaping path is visible in 400 responses — useful for debugging but worth noting for any future security audit.

**Gotchas & invariants:**

- **Lexical normalization only — symlinks are NOT dereferenced.** `path.resolve` is a pure string operation. A symlink inside `ROOT` that points outside `ROOT` will pass the guard; the resolved absolute path of the *symlink itself* is inside `ROOT`, but `readFile` will follow the link and read the target outside `ROOT`. This is a **known security gap** — a standing tech-debt item noted in earlier plan follow-ups. Any future symlink-aware guard must use `fs.realpath` (or equivalent) to resolve the canonical path *after* `path.resolve` and re-validate against `ROOT` before performing any I/O. This would be a breaking change to the module's pure-synchronous contract.
- **`ROOT` is absolute.** `path.resolve` guarantees `ROOT` is absolute regardless of how `FLOWCANVAS_ROOT` was supplied. Callers must supply a relative `rel` (or an absolute one — `path.resolve(ROOT, '/abs/path')` returns `/abs/path`, which will fail the guard unless it happens to be under `ROOT`).
- **No async, no I/O.** `guardPath` is synchronous and pure. It never touches the filesystem. This means it cannot detect whether the resolved path actually exists — callers are responsible for their own `ENOENT` handling.
- **First commit.** `lib/fs-guard.ts` was introduced in commit `3a892ce` (plan 001, phases 1–4, initial architecture). It has not changed since initial creation — the 14-line implementation is stable.

## Known Gaps

- **Symlink traversal not blocked** — `guardPath` uses lexical normalization only (`path.resolve`); a symlink inside `ROOT` pointing outside it will pass the guard and allow the underlying `fs` call to read/write outside `ROOT`. A symlink-aware implementation would require `fs.realpath` post-resolution and re-validation.
