# Templates

- Pure template instantiation library (Decision 8): defines `CanvasTemplate` (board-fragment schema with top-left-relative coords) and exports `instantiateTemplate` — the single entry point every consumer calls to drop a fragment onto the canvas with fresh ids, rebased coordinates, remapped parentIds/edge endpoints, provenance stamps, and uniquified scaffold file paths.

## Purpose

`lib/canvas/templates.ts` owns the pure fragment-cloning logic for the v2 template library. It provides the `CanvasTemplate` board-fragment schema — a named set of nodes, edges, and optional document scaffold files whose node coordinates are stored relative to the fragment's own top-left origin — and the `instantiateTemplate` function that drops an instance of that fragment at a canvas drop point. The function is deliberately pure: it never touches the filesystem, the DOM, or any async I/O. Callers supply a `mint` callback for id-generation so the function stays deterministic in tests. The module is consumed by `lib/canvas/store.ts` (the `addTemplate` Zustand action, which is in turn triggered by `components/canvas/template-tray.tsx` via button and `components/canvas/template-drop.tsx` via drag-and-drop), and by `app/api/templates/route.ts`, which uses `CanvasTemplate` as the shape for `.canvas` fragment files read from the filesystem.

### Internal Architecture

Omitted — single-function module; a graph would restate the source. The flow is: caller builds a `CanvasTemplate` literal or reads one from `/api/templates` → passes to `instantiateTemplate` with a drop point and mint callback → receives `{ nodes, edges, files }` → appends to the canvas doc.

## Usage Examples

```typescript
// Derived from lib/canvas/templates.test.ts:160 — "matches the plan worked example — tpl-x dropped at (500, 300)"

import { instantiateTemplate, type CanvasTemplate } from '@/lib/canvas/templates'

// Deterministic mint factory — each call produces 'n-0001', 'n-0002', 'e-0003', etc.
let i = 0
const mint = (p: string) => `${p}${(++i).toString(16).padStart(4, '0')}`

const tpl: CanvasTemplate = {
  id: 'tpl-x',
  kind: 'diagram',
  name: 'X',
  nodes: [
    { id: 'a', type: 'text', text: 'A', x: 0,   y: 0, width: 100, height: 100 },
    { id: 'b', type: 'text', text: 'B', x: 100, y: 0, width: 100, height: 100, parentId: 'a' },
  ],
  edges: [{ id: 't-e', fromNode: 'a', toNode: 'b' }],
}

const { nodes, edges, files } = instantiateTemplate(tpl, 500, 300, mint)

// nodes[0] → { id: 'n-0001', x: 500, y: 300, meta: { origin: 'user', template: 'tpl-x' } }
// nodes[1] → { id: 'n-0002', x: 600, y: 300, parentId: 'n-0001', meta: { origin: 'user', template: 'tpl-x' } }
// edges[0] → { id: 'e-0003', fromNode: 'n-0001', toNode: 'n-0002', meta: { origin: 'user' } }
// files   → []
```

## Key Insights

**Conventions & patterns:**

- **Pure function contract.** `instantiateTemplate` has no side effects — it neither writes files nor mutates its `CanvasTemplate` input. All I/O (scaffold writes, Zustand state updates) is the caller's responsibility. This keeps the function unit-testable without mocks. (`lib/canvas/templates.ts:1` comment: "Pure: no fs, no DOM".)
- **Caller-supplied mint.** The `mint: (prefix: string) => string` callback is injected by the caller rather than inlined. In tests a deterministic counter is used; in production `crypto.randomUUID().slice(0, 8)` is appended to the prefix. The mint call ordering is stable: all node ids first (iterating `t.nodes` in order), then all edge ids (iterating `t.edges`), then path suffixes (iterating `t.files`). Tests depend on this order — see `lib/canvas/templates.test.ts:117` ("third mint call after two nodes"). (`lib/canvas/templates.ts:33–44`)
- **Coord rebase is additive.** Template node coords are stored relative to (0,0). `instantiateTemplate` adds `dropX`/`dropY` to every node's `x`/`y`. This is the full rebase — there is no bounding-box normalization step. (`lib/canvas/templates.ts:50`)
- **parentId is remapped, not cleared.** Child nodes whose `parentId` is present in the template's node set have it rewritten through `idMap` to the parent's new id. If a `parentId` references a node outside the template (an "orphan" referencing an external group), the original value is preserved unchanged via the `?? n.parentId` fallback — it is never silently dropped to `undefined`. (`lib/canvas/templates.ts:52`)
- **File-path uniquification is a post-close addition.** The `pathMap` / `suffixPath` block (lines 42–57) was added in the "studio UX follow-ups" commit (`0d3244d`) after plan 002 closed, to prevent repeat drops of a `document`-kind template from overwriting the same scaffold file. It inserts a mint-derived suffix before the extension: `dir/scaffold.md` → `dir/scaffold-<sfx>.md`. Critically, the file node's `file` property is kept in lockstep with the new path so the canvas reference and the on-disk file never diverge. (`lib/canvas/templates.ts:19–23`, `lib/canvas/templates.ts:42–57`)
- **Spread-then-override meta pattern.** Both node and edge cloning use `{ ...original.meta, origin: 'user', template: t.id }`. Pre-existing meta fields (e.g., `collapsed`, `shape`, `rel`) are preserved; provenance fields are layered on top. (`lib/canvas/templates.ts:39`, `53–54`)

**Gotchas & invariants:**

- `CanvasTemplate.nodes` coordinates are **relative** to the fragment's own (0,0) top-left, not absolute canvas coords. If a template is authored with absolute coords it will instantiate far off-screen. The convention matches the JSONCanvas "board-fragment" pattern; the `app/api/templates/route.ts` loader validates `nodes` is an array but does not validate coordinate relativity.
- The `mint` callback is called with a bare prefix (`'n-'`, `'e-'`, `''`). When called with `''` (for path suffix generation), the caller should ensure the returned string is non-empty and safe for insertion into a filename; `crypto.randomUUID().slice(0, 8)` satisfies this. A mint that returns an empty string for `''` will produce `suffixPath(p, '')`, which yields `<stem>-.ext` — a valid but odd path.
- Edges whose `fromNode` or `toNode` are not in `idMap` (i.e., they reference nodes outside the template) fall through to `?? e.fromNode` / `?? e.toNode` — the original node id is preserved. This mirrors the `parentId` fallback and prevents silent data loss but may produce dangling edges if the template fragment is used without its full context.
- `files` always returns an array (never `undefined`) — callers can unconditionally iterate it. (`lib/canvas/templates.ts:57`)
- The `mint` call sequence is: (1) all node ids in `t.nodes` order, (2) all edge ids in `t.edges` order, (3) one call per unique `f.path` in `t.files` order. Unit tests pin this sequence (`lib/canvas/templates.test.ts:117`, `138`). Reordering the three blocks would break existing tests.

## Known Gaps

- No runtime validation of `CanvasTemplate.nodes[].x/y` being relative coords — malformed templates (absolute coords) will instantiate silently off-canvas. Manual authoring of `.canvas` fragment files requires care.
- `suffixPath` strips the extension even for extensionless paths (falls to the `else` branch appending `-<sfx>` at the end). Files without extensions are unlikely in practice but are handled correctly.
