# Project Overview

- Flowcanvas renders flowcode markdown files as spatial canvas nodes, connects them with typed/labeled edges, embeds images inline, and round-trips the entire board to/from an AI agent as one self-contained JSON (extended-JSONCanvas 0.5) — now over a native MCP sidecar; a discoverable Agent Generation Kit turns any markdown design doc into a typed system-design `.canvas` with a living bidirectional core spine.
- Architecture: standalone monolith — one Next.js 16 App Router app; client-only React Flow canvas over thin Node-runtime Route Handlers wrapping `fs`; stdio MCP sidecar for harness-native round-trips.
- Primary stack: TypeScript 5 / Next.js 16 / React 19 / React Flow / Zustand / Tailwind v4.
- Datastore: filesystem-backed (`.canvas` JSON files — now schema v0.5, canvas-authoritative); no database, cache, or message bus.
- Modules: 32 subsystems across `lib/canvas/*`, `components/canvas/*`, `app/api/*`, `mcp/` — plan `001` closed 2026-06-27 (10 phases); plan `002` closed 2026-06-29 (7 phases — reopened after runtime testing for Phase 7 defect remediation); plan `004` closed 2026-06-29 (5 phases — Markdown-Core Generation Loop).
- Status active; last updated 2026-06-30 — plans `001` + `002` + `004` + `006-semantic-edges` complete, `005-edges` shipped. Canvas is now the authoritative typed-relation graph with MCP-native agent round-trip, change-review, template library, tri-pane studio shell, closed generation loop, and dot-anchored connection ports; `schemaVersion:'0.5'` is now the persisted version (boards upgrade on open, where the `0.4→0.5` migration seeds a connection port for every edge endpoint and maps legacy `rel → edgeType`); `examples/commerce-system.{canvas,md}` demonstrates the full loop; `examples/commerce-platform.canvas` (now `0.5`, ports seeded) + `templates/*.canvas` exercise every studio surface.

## Project Description

Flowcanvas is a standalone Next.js 16 application for knowledge workers and AI-assisted system designers who need to map, explore, and co-evolve a body of flowcode markdown documents with an AI agent. The canvas renders each `.md` file as a card showing parsed YAML frontmatter and rendered body; it connects nodes with typed/labeled relationships (8-value curated `RelationshipType` catalog), supports agent-creatable groups with provenance, and embeds images inline. The AI agent loop is bidirectional and MCP-native: a stdio sidecar (`mcp/flowcanvas-mcp.ts`) exposes 8 tools (incl. `get_generation_kit`) + the `flowcanvas://generation-kit` resource, wrapping the pure `buildBrief`/`applyResponse`/`buildKit` functions over the guarded HTTP surface; after each round the human reviews, accepts, or discards the full diff via the change-review panel. The `.canvas` file on disk — extended JSONCanvas 0.5 — is the authoritative typed-relation graph; document content stays in live `.md` files. A tri-pane studio shell (structure rail · canvas · inspector) with a template library, bundle export, reference-navigation ↗ chips, and the plan-004 generation loop (kind-typed component widgets + a living core-doc spine) completes the surface.

## Architecture Style

**Standalone monolith + stdio MCP sidecar.** A single Next.js 16 App Router application running in one Node.js process, plus a harness-spawned stdio MCP sidecar (`mcp/flowcanvas-mcp.ts`, run via `tsx`). The canvas screen is a purely client-side React Flow component (loaded with `dynamic({ ssr: false })`) over a thin set of Node-runtime Route Handlers that wrap Node `fs`. There is no external database, no cache, no message bus. The `.canvas` JSON file on disk (schema v0.5) is the authoritative typed-relation graph; `FLOWCANVAS_ROOT` (defaults to `process.cwd()`) governs the guarded filesystem root. The AI agent participates natively over MCP (`FLOWCANVAS_BASE_URL` → HTTP routes) or via clipboard/file JSON round-trip as a fallback. The architecture is intentionally thin: entry layer (`app/page.tsx`), tri-pane studio shell + components (`components/canvas/*`), pure library modules (`lib/canvas/*`), guarded fs routes (`app/api/*/route.ts`), and the MCP sidecar (`mcp/`).

## Code Style & Conventions

- **Language:** TypeScript throughout — no `.js` files, no `any` on contracts.
- **Files:** `kebab-case.ts(x)` everywhere (e.g., `canvas-shell.tsx`, `labeled-edge.tsx`, `fs-guard.ts`).
- **Types / Interfaces / Components:** `PascalCase` (e.g., `CanvasNode`, `FlowcanvasDoc`, `MarkdownNode`).
- **Functions / Variables / Module exports:** `camelCase` (e.g., `nodeKind`, `buildBrief`, `useCanvasStore`).
- **Constants / Sets / Caps:** `UPPER_SNAKE_CASE` (e.g., `BODY_CAP`, `IMAGE_EXT`, `ROOT`).
- **Pure vs. impure split:** `lib/canvas/*` modules are pure TypeScript (no DOM, no React) — they accept typed inputs and return typed outputs. All React + DOM work lives in `components/`. The historical exception was `lib/canvas/adapter.ts`, whose sole job is the React Flow translation boundary; after 005-edges moved per-edge markers into the edge component it imports only type-only symbols from `@xyflow/react` (the former `MarkerType` runtime enum is gone), stays DOM-free, and remains unit-testable under the vitest gate. This boundary is enforced by the vitest gate running only on `lib/canvas/*`.
- **Route handler pattern:** `guardPath(rel)` first; map `GuardError`→`400`, `ENOENT`→`404`, unhandled→`500` via typed `NextResponse.json`.
- **React components:** thin shells over the Zustand store; minimal local state; `memo()` on node components to prevent unnecessary re-renders.
- **Edge IDs:** derived edges use the deterministic pattern `lk:${fromNode}->${toNode}`; agent-minted IDs use `ag-` prefix; user-minted use `e-` prefix.
- See `.flowcode/quality-checks/` sub-files for per-concept rules.
